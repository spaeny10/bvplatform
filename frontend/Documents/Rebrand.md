# Rebrand Guide

Complete rebrand procedure: everywhere "Ironsight" appears to a user, what to change, and what to leave alone.

---

## 1. What counts as a "rebrand"?

This document covers three scenarios. Most of the time you want (a).

| Scenario | What changes | Effort |
|---|---|---|
| **(a) Name-only** | Product name + emails + domain; keep orange/crimson/amber palette | ~15 min |
| **(b) Name + colors** | Above, plus the three brand accent colors | ~30 min |
| **(c) Full rebrand** | Above, plus logo replacement, favicon/PWA icons, documentation sweep | ~60 min |

Every scenario is a one-line-per-location edit. Nothing requires a schema migration, a deployment rebuild (beyond what any code change needs), or downtime.

---

## 2. Architecture overview

Two source-of-truth files drive every user-visible brand string:

```
frontend/src/lib/branding.ts        ← frontend (UI text, logo, CSS vars, PWA manifest)
        │
        ├──→ Logo component              │
        ├──→ PWA manifest                │  frontend
        ├──→ <title>, <meta description> │
        ├──→ --brand-* CSS vars          │
        ├──→ ~15 inline BRAND.name refs  │
        ↓
    user's browser

.env (PRODUCT_NAME=...)              ← backend (evidence-export bundles, future email templates)
        │
        ├──→ cfg.ProductName (Go config struct)
        ├──→ Evidence ZIP README header + footer
        ↓
    downloaded evidence files
```

The two sources are intentionally **duplicated** rather than plumbed through an API:

- The frontend doesn't need a round-trip to know its own product name.
- The backend generates strings in contexts without a request (retention cleanup, async evidence-export workers) where a frontend-sourced value would be awkward.

Keep them in sync by updating both when rebranding — the checklist in §4 walks through this.

### 3.5 Files intentionally left unchanged

Renaming these creates churn with zero user impact. **Don't touch.**

| File | What looks like brand | Why leave it |
|---|---|---|
| [go.mod](../../go.mod) | `module onvif-tool` | Internal identifier; renaming churns every `import` statement |
| Go DB name in [docker-compose.yml](../../docker-compose.yml) (`onvif_tool`) | Matches module name | Renaming requires a DB migration for existing deployments |
| Frontend file paths (`@/lib/ironsight-api`, `@/types/ironsight`) | 80+ hits | Internal module names; never seen by users |
| Component import paths (`@/components/shared/Logo` — but inside it was `IronsightLogo`) | Already renamed | Done |
| `localStorage` keys (`ironsight_token`) | 10+ hits | Renaming logs every existing user out |
| Log prefixes (`[ONVIF]`, `[REC]`, etc.) | Internal | Operator-facing diagnostic prefixes |
| Database table names, column names | Internal | Schema change is expensive |
| [src/components/operator/EvidenceExportModal.tsx](../src/components/operator/EvidenceExportModal.tsx) print template | Uses `'Segoe UI'` + hardcoded dark-on-white colors | It's a print-optimised HTML template injected into a print dialog — intentionally themed differently from the app |

## 4. The checklist

### 4.1 Scenario (a) — Name-only rebrand

Say the new name is **SkyWatch**.

```
□ 1. Edit frontend/src/lib/branding.ts:
     name:         'SkyWatch',
     shortName:    'SkyWatch',
     tagline:      'Intelligent Site Surveillance',
     description:  'Real-time …',   ← rewrite for SEO / PWA description
     homeUrl:      'https://skywatch.example.com',
     supportEmail: 'support@skywatch.example.com',
     noreplyEmail: 'noreply@skywatch.example.com',

□ 2. Edit .env (or wherever PRODUCT_NAME is set in your deployment):
     PRODUCT_NAME=SkyWatch

□ 3. Find-replace seed emails in cmd/server/main.go:
     @ironsight.io  →  @skywatch.example.com   (6 occurrences)
     Only matters on fresh-DB first-boot; skip on existing deployments.

□ 4. Rebuild + restart:
     frontend:  npm run build     (Dockerfile does this in container)
     backend:   go build ./cmd/server

□ 5. Hard-refresh the browser. Browser-cached HTML still has old title.

□ 6. Verify in the UI:
     - Tab title says "SkyWatch"
     - Login page logo wordmark renders "SkyWatch"
     - PWA icon still has old art (§4.3 if you want to fix)
     - Evidence download README says "SKYWATCH EVIDENCE EXPORT"
```

~15 minutes end to end. No schema changes, no migration.

### 4.2 Scenario (b) — Name + colors

Everything in (a), plus:

```
□ 7. Edit BRAND.colors in branding.ts:
     colors: {
       primary:   '#0066CC',    // was '#E8732A'
       secondary: '#00447A',    // was '#B22234'
       tertiary:  '#4FA6E3',    // was '#E89B2A'
     }

□ 8. (Optional) If the portal's warm-cream light theme doesn't harmonise
     with the new brand, edit --accent in portal.css light block:
     frontend/src/app/portal/portal.css line ~17
     See Documents/HOUSEKEEPING.md §2.4 for the portal palette rationale.

□ 9. Hard-refresh. Every CTA, active tab, focus ring, logo dot now
     reads in the new brand color — because they all derive from
     --brand-primary via the alias chain in globals.css.
```

### 4.3 Scenario (c) — Full rebrand with assets

Everything in (b), plus:

```
□ 10. Prepare the assets. Required sizes:
      - favicon.ico   — 32×32 multi-res (16, 32, 48 all embedded)
      - icon-192.png  — 192×192 PWA icon, transparent or matched background
      - icon-512.png  — 512×512 PWA icon, same
      - logo.svg      — optional; if you want a full custom logo instead of
                        the auto-generated wordmark

□ 11. Place them in frontend/public/:
      - favicon.ico
      - icon-192.png
      - icon-512.png
      - logo.svg     ← optional

□ 12. If using a custom logo, set BRAND.logoSrc in branding.ts:
      logoSrc: '/logo.svg',
      The <Logo> component falls back to <img> instead of the generated SVG.

□ 13. Find-replace in all documentation:
      frontend/Documents/*.md
      (Ironsight → SkyWatch, IRONSight → SKYWATCH)
      Prose, not structured data — a plain find-replace is safe.

□ 14. Rebuild Docker images (they embed the frontend + backend artifacts):
      docker compose build --no-cache

□ 15. Deploy. Users may need to uninstall/reinstall any PWA installs to
      pick up the new icons (browser cache on the manifest).
```

## 5. Walkthroughs — the non-obvious bits

### 5.1 Fixing the seed emails in cmd/server/main.go

These emails are only written to the DB on first-boot of a fresh install — they become rows in the `users` table. Existing deployments keep whatever was seeded when they first started.

```bash
# One-line fix, safe to run on your source:
sed -i 's/@ironsight.io/@skywatch.example.com/g' cmd/server/main.go
go build ./cmd/server
```

If the deployment is already running, the emails in the DB are now historical — change them through the admin UI (Users tab) or via SQL.

### 5.2 Frontend CSV download filenames

The portal and audit-log export both generate filenames like `ironsight-audit-log-2026-04-24.csv`. These are wired to `BRAND.shortName`:

```ts
// frontend/src/components/admin/AuditLogExport.tsx
const DOWNLOAD_SLUG = BRAND.shortName.toLowerCase().replace(/\s+/g, '-');
// `${DOWNLOAD_SLUG}-audit-log-${date}.csv`
```

A name like "Sky Watch" (with a space) becomes `sky-watch-audit-log-2026-04-24.csv`. No action needed.

### 5.3 Backend evidence-export README

The README inside every evidence ZIP has a branded header + footer:

```
SKYWATCH EVIDENCE EXPORT
========================================

[...event data...]

Generated by SkyWatch Platform. This package is a faithful
copy of recorded data at the time of export.
```

This uses `cfg.ProductName` from `PRODUCT_NAME` env. When the rebuild rolls out, new exports pick up the new name; old exports are frozen with whatever header they were written with (that's the point of an export — the artifact is immutable).

### 5.4 Docker container names

The compose file uses `container_name: ironsight-api`, etc. To rename all six containers, set `COMPOSE_PROJECT_NAME=skywatch` in `.env`. Compose will prefix container names with the project — on fresh deploys, but on existing ones it'll create *new* containers alongside the old ones. To fully rename:

```bash
docker compose down
# edit .env: COMPOSE_PROJECT_NAME=skywatch
docker compose up -d
```

No data lost — volumes persist across the rename because they're declared at the top level.

### 5.5 White-label / per-customer branding

Not supported. The rebrand described here is a single global rebrand — every customer sees the same product name. Per-customer white-labeling would require multi-tenant brand resolution (org_id → brand config), which is intentionally out of scope. If a customer asks for it, scope it as a separate project — the easy places to hook in would be layout.tsx (reading org from JWT claim) and a new `brand_overrides` DB table.

## 6. Testing the rebrand

After any edit to branding.ts:

```bash
cd frontend
npx tsc --noEmit              # should pass
npx next build                # should pass
```

Smoke-test the surfaces that show brand:

1. **Tab title** — `http://localhost:3000/` → tab should read "NewName — Tagline"
2. **Login page logo** — wordmark SVG renders the new name
3. **Analytics footer** — "NewName Analytics · N sites"
4. **Evidence viewer** — `/evidence/expired` → "Powered by NewName"
5. **Admin → Audit Trail** — footer says "NewName Audit Trail"
6. **Settings → Build** — "NewName Platform"
7. **Download a CSV** from the portal — filename starts with `newname-`
8. **Download an evidence bundle** from the operator console — README header says "NEWNAME EVIDENCE EXPORT"
9. **PWA manifest** — visit `/manifest.webmanifest` → JSON shows new `name`, `short_name`, `description`, `theme_color`
10. **CSS brand cascade** — DevTools → `<html>` element → Computed styles should show `--brand-primary` = new hex; every CTA button reflects the new color

If any of these still show the old brand, grep the repo for the old name and check against the "Files intentionally left unchanged" list in §3.5 — if it's in that list, it's expected.

---

## 7. Rollback

A rebrand is purely additive — no schema changes, no data migration. Rollback is a git revert of the branding.ts + .env changes. The backend's evidence-export PRODUCT_NAME change takes effect on the next deploy; any ZIP bundles written during the rebranded period keep the new name forever (immutable artifacts, as designed).

If a rebrand ships with a bug (say, the new color fails WCAG contrast on an important surface), rollback is safe at any time; no need to undo data.
