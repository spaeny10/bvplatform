#### F-13 — SSRF: proxy upstream built from unvalidated `onvif_address`, no allow/deny-list
`internal/api/camera_web_proxy.go:103-114,147` · **dimension: security**

`upstream := &url.URL{Scheme:"http", Host: cam.OnvifAddress, …}` then `client.Do(req)`, with no check against loopback, link-local, `169.254.169.254`, or RFC1918-internal ranges. Verifier traced full reachability: the route requires only `RequireAuth` + `CanAccessCamera` (no admin role), the response body is returned (read-SSRF, not blind), and the destination is attacker-controlled because camera create/update accepts any non-empty `onvif_address` (F-14). Real internal targets: AI sidecars (`127.0.0.1:8501/:8502`), mediamtx, and `/metrics` whose production model is `METRICS_AUTH=none` network-trust — an SSRF from the app's own loopback bypasses the NPM IP restriction. Also the exfiltration channel for F-12's cookie.
**Fix:** validate `onvif_address` against the trailer-LAN allow-list (e.g. `192.168.50.0/24`), reject loopback/link-local/internal ranges, and resolve-and-pin the host before `client.Do` (DNS-rebinding guard).
- [ ] approve fix

#### F-15 — Login rate-limit key and failed-login audit trust client-supplied `X-Forwarded-For`
`internal/api/audit_playback.go:70` · `ratelimit.go:81`, `auth_handler.go:190` · **dimension: security**

`clientIP()` reads `X-Forwarded-For` **first** and returns the left-most value before falling back to `RemoteAddr`. `RateLimitLogin` keys its 10/min bucket on `clientIP(r)`, and `logFailedLogin` records it as the forensic IP. chi's `middleware.RealIP` also blindly trusts left-most XFF and doesn't strip the header, so the value is attacker-controlled even behind NPM (which appends, leaving the spoofed value left-most). Rotating XFF per request resets the bucket every time, defeating the only password-spray defense (account lockout is per-username and doesn't stop spraying across accounts), and poisons the failed-login audit with forged source IPs — defeating the documented UL 827B brute-force control.
**Fix:** derive the rate-limit/audit IP from the real transport peer (trusted-proxy-aware); only honor XFF when the immediate peer is a configured trusted proxy and take the right-most untrusted hop. Never consult raw XFF first for security keys.
- [ ] approve fix

#### F-21 — Camera web-UI proxy `<base href>` cannot reroute root-relative URLs — the injection's stated purpose doesn't hold
`internal/api/camera_web_proxy.go:272` · **dimension: correctness**

`injectBaseHref` injects `<base href="/api/cameras/<id>/web-ui/">` claiming it makes "absolute-rooted relative URLs (like `/static/main.js`) resolve back through the proxy." Per RFC 3986 §5.3 / WHATWG URL resolution, a path-absolute reference takes only scheme+authority from the base and its path **replaces** the base path — so `/static/main.js` resolves to `https://<ironsight-host>/static/main.js`, bypassing the proxy and 404/401-ing on the Ironsight origin. `<base>` only affects document-relative references; nothing rewrites root-relative URLs in HTML/JS/CSS bodies. Cameras whose firmware emits root-relative asset/XHR/form URLs render broken in the iframe. Works for document-relative vendors; an "open in new tab" fallback exists.
**Fix:** rewrite root-relative URLs in the HTML body to the proxy prefix, or serve the iframe from a dedicated subdomain/path with a catch-all that forwards unmatched root paths to the camera.
- [ ] approve fix

#### F-22 — Stall watchdog is blind in event mode (and for cameras with no completed segment)
`internal/recording/engine.go:748` · `clip_writer.go:136` · **dimension: correctness**

The watchdog `os.ReadDir(r.outputDir)` non-recursively and sums only top-level `*.mp4`, with a kill condition gated on `!latestMod.IsZero()`. In event mode ffmpeg writes ring segments to `outputDir/ring` and `ClipWriter` writes clips into `outputDir/<clipID>/` subdirectories, so `outputDir` has no top-level `.mp4` → `latestMod` stays zero → the watchdog can never fire. There are no ffmpeg `-rw_timeout`/`-stimeout` flags, so a silent cellular TCP stall blocks `cmd.Wait()` indefinitely; with no restart and no `ffmpeg_subprocess_crash` alert, the ring stops filling and subsequent event clips are empty. Brand-new cameras that hang before the first segment lands fail the same way. `/api/recording/health` is pull-only and can't distinguish "stalled" from "no events."
**Fix:** in event mode point the watchdog at `r.ringDir`, and treat "no `.mp4` has ever appeared after N minutes since Start" as a stall.
- [ ] approve fix

#### F-26 — Data race on `Recorder.cmd` (written in `runFFmpeg` without the mutex, read by `Stop()` and the watchdog)
`internal/recording/engine.go:672` · **dimension: correctness (downgraded medium → low)**

`r.cmd = exec.CommandContext(...)` runs on the recordLoop goroutine with no lock and is reassigned on every restart; `Stop()` reads it under `r.mu` and the watchdog reads it with no lock — a genuine `-race`-flaggable data race (CI runs `-race` but no test exercises it). **Corrected impact:** the claimed "orphaned ffmpeg holding an RTSP session" is refuted — `Stop()` calls `r.cancel()` on the same ctx passed to `exec.CommandContext`, so any ffmpeg under that ctx is killed regardless of which stale pointer `Stop` observes; killing an already-waited `Cmd` is benign. Worst realistic effect: a dying watchdog kills a freshly-restarted ffmpeg once (self-heals next loop) plus theoretical memory-model UB.
**Fix:** guard `r.cmd` assignment with `r.mu` (or `atomic.Pointer[exec.Cmd]`); have the watchdog capture the specific cmd for its run.
- [ ] approve fix

#### F-27 — `ClipWriter` clears `recording` before copying ring segments — window for `CleanRingBuffer` to delete the pre-buffer
`internal/recording/clip_writer.go:116` · **dimension: correctness**

`writeClip` clears `cw.recording=false` under the mutex, **then** does the per-segment copy loop. `CleanRingBuffer` (every 5 s) skips only while `recording` is true and deletes ring files older than `now-(preBufferSec+5)`; with defaults (pre 10 / post 30) the pre-buffer segments are already past the cutoff the moment the flag flips. If the 5 s tick lands during the ~hundreds-of-ms copy loop, pre-buffer segments are deleted before `collectRingSegments` reads them — the ENOENT branch silently skips them while the DB row still claims the full duration, producing event clips missing the lead-up footage. Probabilistic per clip, with a log signal.
**Fix:** collect and copy the ring segments first, then clear `cw.recording` (or use a separate "finalizing" flag that `CleanRingBuffer` also respects).
- [ ] approve fix

#### F-28 — CSRF exemption for the web-ui proxy is a method-blind path-substring match over all `/api`
`internal/api/auth_handler.go:125` · **dimensions: correctness + security (merged, 2 finders; downgraded medium → low)**

`if strings.Contains(r.URL.Path, "/web-ui/") || strings.HasSuffix(r.URL.Path, "/web-ui")` exempts any mutating method on any `/api/*` path containing that substring, before route matching, instead of being anchored to the chi route pattern. **No current exploit:** the only paths containing `/web-ui/` are the intentionally-exempt proxy routes (still behind `RequireAuth`), chi `{param}` segments can't contain a slash, and `SameSite=Lax` on the session cookie is the working primary CSRF defense. Forward-looking fragility only (a future wildcard route containing the substring would silently lose CSRF protection).
**Fix:** exempt based on the matched chi `RoutePattern` (or route-level middleware composition), scoped to the proxy route only.
- [ ] approve fix

#### F-29 — Rewritten camera `Set-Cookie` loses `Secure`/`SameSite`; `Path` scoping is not an isolation boundary
`internal/api/camera_web_proxy.go:128` (`rewriteSetCookie` ~`:220`) · **dimension: security**

`rewriteSetCookie` strips `Domain` and forces `Path=/api/cameras/<id>/web-ui/` but never sets `Secure` or `SameSite`, and the cookies land first-party on the Ironsight HTTPS origin. The commit's "isolated per camera path" claim overstates it — `Path` is not a security boundary; content through the proxy is same-origin with the rest of the app. Modest weakness (route behind `RequireAuth` + `CanAccessCamera`, camera cookies low-value); the deeper same-origin concern is covered by F-12/F-13.
**Fix:** force `Secure` (when `CookieSecure`) and an explicit `SameSite` on rewritten cookies; document that camera-UI sessions share the Ironsight origin, or serve the proxy from a sandboxed subdomain if true isolation is required.
- [ ] approve fix

#### F-30 — mediamtx config uses deprecated `readBufferCount`; comment misdescribes it; unpinned `:latest` risks future startup failure
`internal/streaming/mediamtx.go:460` · `docker-compose.yml:79` · **dimension: correctness**

The field comment says `ReadBufferCount: 2048` "sizes mediamtx's per-source RTP reorder buffer," but in mediamtx 1.x this key is `deprecated:"true"` and maps to `WriteQueueSize` (the outgoing-packet write queue) with a deprecation warning — so the comment is wrong and the setting only works via a deprecated alias. Because the image is `:latest` (unpinned) and mediamtx rejects unknown config keys, a future pull that drops the alias makes the generated `mediamtx_runtime.yml` fatal → crash-loop, taking down RTSP relay (recording ingest) and native HLS (live view). Works today; the outage is latent and version-contingent.
**Fix:** rename the field/yaml key to `writeQueueSize`, correct the comment, and pin the mediamtx image to a specific version.
- [ ] approve fix

#### F-31 — `applyCameraVCAToDB` deletes all rules then re-creates without a transaction — mid-way failure wipes the rule set
`internal/api/vca_pull.go:461` · **dimension: correctness**

The handler loops `db.DeleteVCARule` (hard delete) for every existing rule, then loops `db.CreateVCARule`; any create error returns immediately with deletions already committed and no rollback. A `POST /vca/pull?apply=1` that fails on rule N leaves the camera with zero or partial platform-side VCA rules — the AI detection pipeline silently loses its zones. Narrow: requires a mid-loop DB failure during an admin-initiated apply; surfaced as a 500, and a successful re-run restores the set (the camera is the source of truth). The pull/sync UI is currently behind a `false &&` guard (`VCAZoneEditor.tsx:600`), but the endpoint is still routed and admin-callable.
**Fix:** wrap delete+create in a single pgx transaction (rollback on error), or insert new rules first and delete the old set only after all creates succeed.
- [ ] approve fix
