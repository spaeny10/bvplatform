# Authenticated, tenant-scoped media serving (P1-A-03)

## What this fixes

Before this change, the Ironsight server published three bare
`http.FileServer` routes:

| Route          | Disk path                                     |
| -------------- | --------------------------------------------- |
| `/recordings/*`| `cfg.StoragePath` (continuous MP4 segments)  |
| `/hls/*`       | `cfg.HLSPath`     (live HLS playlists + .ts)  |
| `/snapshots/*` | sibling of `StoragePath` (alarm JPEGs)        |

All three were unauthenticated. Anyone who could guess (or otherwise
obtain) a camera UUID + filename could fetch the bytes â€” even a
logged-in customer from a different organisation. UL 827B and any
serious customer security review would flag this as a hard fail.

## What changed

- Three FileServer routes are gone.
- New route: `GET /media/v1/{token}`. Public (no auth middleware); the
  token IS the authorisation. The handler validates it, re-checks
  tenant scope against the live DB, then streams the file.
- New route: `POST /api/media/mint`. JWT-authenticated; returns a
  short-lived signed URL bound to one specific file.
- Per-segment / per-snapshot / per-playlist URLs are now opaque tokens.
  Cross-tenant URL-guessing is no longer a vector.

## The eight design decisions

1. **Token transport: URL only.** JWT sits in the path
   (`/media/v1/<token>`), not a cookie, not a header. Browsers happily
   put any URL into `<video src>` and `<img src>` without ceremony,
   which is what we need for HLS players, snapshot thumbs, and
   evidence-export links. The token is the unit of authorisation.

2. **TTL: default 5 min, ceiling 1 h.** Most surfaces (live HLS, the
   operator console's alarm snapshots) refresh well within 5 min.
   Evidence-export bundle download links want a longer window so the
   recipient can open the email + click the link the next day; the
   mint endpoint accepts an optional `ttl_seconds` capped at 3600s for
   that case. Anything beyond requires re-minting.

3. **HLS: dynamic playlist rewriter.** The on-disk `.m3u8` files
   (`main_live.m3u8`, `sub_live.m3u8`, the VOD playlist generated by
   `/api/playback/{id}/playlist.m3u8`) reference segments by relative
   filename. On every fetch through `/media/v1/<token>` where the
   token's `path` is a `.m3u8`, the handler reads the playlist line by
   line and rewrites each URI line to a freshly-minted child
   `/media/v1/<sub-token>`. Comment lines (`#EXT-X-*`) pass through
   byte-identical. The `URI="..."` attribute on `#EXT-X-MAP` is also
   rewritten in-place. Cost: ~1 ms per playlist re-read; HLS playlists
   are small (â‰¤ 5 segments live, â‰¤ 60 segments VOD).

4. **Cross-tenant: 404, not 403.** The 404 hides whether the camera
   even exists in another org. 403 leaks "yes, that UUID is real" â€”
   which is enough information for an attacker to start mapping the
   tenant graph. This is the security-critical case; the test
   `TestMediaServe_CrossTenantDenied_Returns404` is the
   commit-must-block assertion.

5. **Re-verify scope on every serve.** The token claim says
   "user U may fetch camera C's file P" â€” but a role change between
   mint and serve must take effect immediately. We hit the DB once per
   serve: load the user's current role, walk the cameraâ†’siteâ†’org
   chain, call `CanAccessCamera`. ~0.5 ms per serve on indexed
   columns; well under the network round-trip. Without this, demoting
   a customer would leave their 5-min-old tokens valid for 5 more min.

6. **Lockstep rollout.** No feature flag. Backend + frontend land in
   the same commit (or commit series, all on `sso-patches`). The
   three old FileServer routes are deleted; the new
   `/media/v1/{token}` is the only path. Frontend surfaces that
   previously hardcoded `/hls/...` / `/snapshots/...` paths either
   call `mintMediaToken` directly (VCAZoneEditor, portal site
   drilldown), pass the legacy URL through `HLSVideoPlayer` (which
   detects + transparently re-mints), or render via the new
   `<SignedImage>` shared component.

7. **Audit: ring buffer, batched insert.** A synchronous DB write per
   serve would tank the hot path (4 cams Ã— 30 segments/min â‰ˆ 245k
   rows/day). Every serve enqueues one row into a per-router
   in-memory ring buffer; a background goroutine flushes 100-row
   batches into `audit_log` every 5 sec (or sooner if the ring
   fills). Failure mode: a server crash drops at most ~5 sec of audit
   rows. That's an acceptable trade â€” the playback_audits and
   recording-segment trail are the courtroom record; the audit_log
   "who-fetched-what" entries here are operational forensics, not
   compliance evidence.

8. **Out-of-band tools: same path.** Anyone wanting a recording â€” the
   evidence-export bundle path, ML-labeling annotators, future
   integration partners â€” mints via `/api/media/mint` and uses the
   returned `/media/v1/<token>`. There is no backdoor; the URL scheme
   is the API.

## Token format

JWT, HS256, signed with `Config.JWTSecret` (the same secret as session
tokens â€” audience-isolated via the `iss` claim).

```json
{
  "sub":  "<user_id>",
  "cam":  "<camera_uuid>",
  "path": "seg_20260511_120000.mp4",
  "kind": "segment" | "hls" | "snapshot",
  "iss":  "ironsight-media-v1",
  "exp":  <unix-ts-5min-from-now-or-mint-supplied>,
  "iat":  <unix-ts-of-mint>,
  "jti":  "<uuid-v4>"
}
```

Field rules enforced at both mint and parse time:

- `kind` must be one of `segment` / `hls` / `snapshot`.
- `path` must match `[a-zA-Z0-9._-]+`, not start with `.`, â‰¤ 255 chars.
  Mint *and* parse both refuse anything else.
- `iss` must be exactly `ironsight-media-v1`. Session JWTs have a
  different (empty default) issuer and are rejected by `ParseMediaToken`
  â€” and vice versa via the typed-claims split.

## URL scheme

```
/media/v1/<JWT>
```

Pure path. No query string. No `Authorization` header. No cookies.
This is the only shape â€” the absence of any extra plumbing keeps the
URL safe to embed in `<video src>` / `<img src>` / mail attachments.

The mint response is:

```json
{
  "url":        "/media/v1/eyJhbGc...",
  "expires_at": "2026-05-11T12:05:00Z"
}
```

## Audit format

One row per successful serve, written via a 100-row batched INSERT
into `audit_log` every 5 sec (or sooner on ring-full):

| Column        | Value                                          |
| ------------- | ---------------------------------------------- |
| `user_id`     | The mint-time subject (re-resolved from DB)    |
| `username`    | The user's current username                    |
| `action`      | `media_serve`                                  |
| `target_type` | `camera`                                       |
| `target_id`   | Camera UUID                                    |
| `details`     | `{"kind":"segment","path":"seg_...mp4"}` JSON  |
| `ip_address`  | `clientIP(r)` (X-Forwarded-For-aware)          |
| `created_at`  | Wall-clock at serve time (UTC)                 |

The `(target_type, target_id, created_at DESC)` index already on
`audit_log` covers "show me every fetch of camera X in the last 24 h"
queries without a table scan.

## Frontend integration

| Surface                               | Pattern                                |
| ------------------------------------- | -------------------------------------- |
| Live HLS player (operator + admin)    | `<HLSVideoPlayer src="/hls/<cam>/main_live.m3u8" />` â€” the player auto-re-mints. |
| VCA snapshot fallback                 | `mintMediaToken({ kind: 'hls', ... })` directly, one-shot. |
| Portal site-drilldown snapshot        | `mintMediaToken({ kind: 'snapshot', ... })` then fetch as blob. |
| `<img src={alarm.snapshot_url}>` etc. | Replace with `<SignedImage src={...} />` â€” handles legacy + signed URL shapes. |
| Recording playback (event modal)      | Backend already returns signed URL in `event.playback_url`. No frontend change. |

The shared `<SignedImage>` and `HLSVideoPlayer` components both call
`resolveMediaURL` (in `frontend/src/lib/media.ts`), which understands
all four URL shapes (legacy `/recordings/`, `/hls/`, `/snapshots/`,
already-signed `/media/v1/`). New code should pass through these
shared components instead of constructing media URLs by hand.

## Risks already addressed in code

- **Path traversal.** Allow-list at mint time *and* allow-list at
  parse time. `filepath.Base()` applied a third time at resolve time
  as defence-in-depth. Tested with `TestSignMediaToken_RejectsBadPath`.
- **HLS player compatibility.** Tokens go in the path, not a query
  string â€” Safari's native player handles relative-URI playlist
  rewrites cleanly. fMP4 init-segment URIs in `#EXT-X-MAP` are
  rewritten in-place.
- **JWT secret rotation.** When `JWT_SECRET` rotates, live media
  tokens become invalid. 5-min TTL means the next mint succeeds; any
  in-flight playback gets a 401 on its next segment fetch and the
  player re-fetches the playlist (which mints fresh tokens). No
  manual coordination needed.
- **Audit ring overflow.** Drop-on-full, log a warning. Better to
  miss an audit row than block the hot path. Sustained burst > 200
  serves/sec for >5 sec would lose rows; current peak (4 cams Ã—
  30 seg/min) is ~2 serves/sec.

## Out of scope here

- `/exports/*` (evidence-export bundle download). Still served by

- `/webrtc/*` (MediaMTX WHEP proxy). Unchanged â€” WebRTC has its own
  auth model and isn't a customer-media-file path.
